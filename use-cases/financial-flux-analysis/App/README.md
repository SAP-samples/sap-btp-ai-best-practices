## Overview

Financial Flux Analysis is a full-stack financial intelligence application built on SAP BTP. It ingests transactional journal entry data from SAP HANA / SAP Datasphere, performs year-over-year variance analysis entirely in the backend, and presents the results through an SAP Fiori interface with AI-generated executive narratives.

**Key design principle:** SAP HANA returns raw data only. All aggregations, joins, enrichments, and calculations are performed in Python — never in HANA SQL.

## Features

### Financial Analysis
- **Year-over-year variance analysis** — compare any two fiscal years across all dimensions
- **Multi-dimensional grouping** — analyze by G/L Account, Profit Center, Cost Center, Segment, Functional Area
- **Monthly drilldown** — month-by-month breakdown for any selected dimension key
- **Flash expense analysis** — categorized expense view using SAP `v_glaccount` Flash Line Item classification
- **WBS (Work Breakdown Structure) analysis** — project-level financial tracking

### AI-Powered Insights
- **AI Explain** — per-account LLM narrative generated by SAP AI Core (GPT-4 class models)
  - Accounting-aware variance direction (favorable/unfavorable based on account nature)
  - Netting analysis — detects and explains internal RC/CO/AB reallocation entries
  - CONCUR employee expense detection (not classified as procurement)
  - Supplier attribution from SAP fields only — no hallucination
  - Flash Line Item narrative using SAP master data
- **Rule-based fallback** — if LLM is unavailable, a structured summary is returned automatically
- **Conversational Chat** — context-aware financial Q&A powered by SAP AI Core

### Data Integrity
- Schema alignment before UNION ALL — missing optional columns auto-filled as NULL
- Row count integrity check after every union operation
- Strict schema validation post-alignment
- No silent failures — all schema issues are explicitly logged

### Filters
- Company Code
- Segment
- Functional Area
- Fiscal Year (current + previous)

## Data Sources

All views are loaded from SAP HANA schema `COEAI`. View names are configurable via environment variables.

| Environment Variable | Default Value | Description |
|---|---|---|
| `HANA_VIEW_BASEDATA` | `v_basedata` | Primary transactional journal entries |
| `HANA_VIEW_NEWCC` | `v_newcc` | Additional transactional records (new cost centers) |
| `HANA_VIEW_CECO1711` | `ceco1711` | Additional transactional records (cost center 1711) |
| `HANA_VIEW_PROFITCENTER` | `v_profitcenter` | Profit center master data |
| `HANA_VIEW_COSTCENTER` | `v_costcenter` | Cost center master data |
| `HANA_VIEW_GL_GROUPING` | `v__glaccountgrouping` | G/L account groupings |
| `HANA_VIEW_GLACCOUNT` | `v_glaccount` | Authoritative G/L account master + Flash Line Item |

## Backend

### Ingestion Pipeline

Located in `backend/loaders/raw_loader.py`.

```
1. load_basedata()     → SELECT * FROM "v_basedata"
2. load_newcc()        → SELECT * FROM "v_newcc"
3. load_ceco1711()     → SELECT * FROM "ceco1711"
4. _align_schema()     → detect missing columns → add as NULL
5. _union_transactional() Step 1: v_basedata UNION ALL v_newcc
6. _union_transactional() Step 2: (combined) UNION ALL ceco1711
7. Integrity check: combined_count == sum(source_counts)
```

**Schema alignment rules:**
- Missing columns in any source are added explicitly as `pd.NA`
- Every auto-added column is logged individually
- Strict validation runs after alignment — any remaining incompatibility raises `SchemaCompatibilityError`
- No silent continuation permitted

### Enrichment Pipeline

Located in `backend/enrichers/dataset_enricher.py`.

The combined transactional DataFrame is enriched via LEFT JOINs:
- `v_profitcenter` → Profit Center hierarchy and segment mapping
- `v_costcenter` → Cost Center hierarchy
- `v__glaccountgrouping` → G/L account grouping and Flash Line Item
- `v_glaccount` → Authoritative Flash Line Item, FS Item, account nature (EXPENSE/REVENUE/ASSET/LIABILITY)

The enriched dataset is cached in memory by `FinancialProcessor` and reused across all API requests.

### API Endpoints

| Method | Endpoint | Description |
|---|---|---|
| `GET` | `/health` | HANA connectivity check + cached row count |
| `GET` | `/api/init-data` | Available fiscal years, company codes, segments, functional areas |
| `GET` | `/api/grouped-analysis` | FY vs FY aggregation by dimension |
| `GET` | `/api/group-detail` | Monthly breakdown for a specific dimension key |
| `GET` | `/api/account-line-items` | Raw journal entry line items for a key |
| `POST` | `/api/ai-explain` | AI-generated variance narrative for a specific account |
| `POST` | `/api/refresh` | Force reload of all data from HANA |
| `POST` | `/api/chat` | Conversational financial Q&A |

#### `/api/grouped-analysis` parameters

| Parameter | Type | Description |
|---|---|---|
| `group_by` | string | `G/L Account`, `Profit Center`, `Cost Center`, `Segment`, `Functional Area` |
| `current_year` | int | Current fiscal year |
| `previous_year` | int | Comparison fiscal year |
| `company_code` | string (optional) | Filter by SAP company code |
| `segment` | string (optional) | Filter by segment |
| `functional_area` | string (optional) | Filter by functional area |

### AI Analysis

The `/api/ai-explain` endpoint builds a structured prompt containing:

1. **Account metadata** — key, name, account nature from SAP master data
2. **Year-over-year summary** — totals, delta, delta %
3. **Monthly breakdown** — 12-month comparison
4. **G/L account breakdown** — for Profit Center / Cost Center analysis
5. **Document type breakdown** — SAP journal entry types with amounts
6. **Netting analysis** — detects RC/CO/AB internal reallocation entries
7. **Supplier attribution** — from SAP supplier fields only
8. **Flash Line Item analysis** — from `v_glaccount` master data
9. **Accounting context** — favorable/unfavorable direction based on account nature

The prompt is sent to SAP AI Core (via `sap-ai-sdk`). If the LLM call fails, a rule-based fallback summary is returned automatically.

**Supported document type classifications:**
- Reallocation types: `RC`, `CO`, `AB`, `SA`, `JE`, `KP`
- External/procurement types: `KR`, `RE`, `WE`, `KG`, `KZ`
- CONCUR employee expense detection via journal entry text patterns

### Chat Module

Located in `backend/src/chat/`.

- Context-aware conversational interface for financial data queries
- Maintains conversation history per session
- Uses `FinancialProcessor` data as context
- Routes through SAP AI Core LLM
- Registered as an additive FastAPI router — does not modify existing endpoints

## Frontend

Built with **SAP UI5 / OpenUI5** and bundled with **Vite**.

### Views

| View | Description |
|---|---|
| `Variances` | Main variance analysis screen — grouped FY vs FY comparison with AI explain dialog |
| `WBS` | Work Breakdown Structure financial analysis |
| `Chat` | Conversational AI financial assistant |

### Key Components

- **Variance table** — sortable, filterable by company code / segment / functional area / fiscal year
- **AI Analysis Dialog** — per-row LLM narrative with Flash expense breakdown
- **Monthly drilldown chart** — triggered from any variance row
- **Line items table** — raw journal entries for any selected account
- **Chat interface** — streaming conversational Q&A

### API Service

`fiori/webapp/services/apiService.js` — centralized HTTP client for all backend calls. All frontend screens consume this service; no direct HANA access from the frontend.

## Configuration

Copy `backend/sample.env` to `backend/.env` and fill in your values:

```env
# SAP HANA Connection
HANA_ADDRESS=<your-hana-host>.hanacloud.ondemand.com
HANA_PORT=443
HANA_USER=<your-user>
HANA_PASSWORD=<your-password>
HANA_ENCRYPT=True
HANA_SCHEMA=COEAI

# SAP HANA Source Views
HANA_VIEW_BASEDATA=v_basedata
HANA_VIEW_NEWCC=v_newcc
HANA_VIEW_CECO1711=ceco1711
HANA_VIEW_PROFITCENTER=v_profitcenter
HANA_VIEW_COSTCENTER=v_costcenter
HANA_VIEW_GL_GROUPING=v__glaccountgrouping
HANA_VIEW_GLACCOUNT=v_glaccount

# SAP AI Core (GenAI Hub)
AICORE_AUTH_URL=<oauth-token-url>
AICORE_CLIENT_ID=<client-id>
AICORE_CLIENT_SECRET=<client-secret>
AICORE_BASE_URL=<ai-core-base-url>
AICORE_RESOURCE_GROUP=<resource-group>
```

> **Never commit `.env` to version control.** Use `sample.env` as the template.

## Local Development

### Backend

```bash
cd backend
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp sample.env .env   # fill in your credentials
uvicorn api:app --reload --port 8000
```

API available at: `http://localhost:8000`  
Interactive docs: `http://localhost:8000/docs`

### Frontend

```bash
cd fiori
npm install
npm run dev
```

Frontend available at: `http://localhost:5173`

> The Vite dev server proxies `/api` requests to `http://localhost:8000` by default.

## Deployment

Both components are deployed to **SAP BTP Cloud Foundry**.

### Backend

```bash
cd backend
cf push
```

Uses `manifest.yml` for CF configuration. Environment variables must be set via CF environment or a bound service — **not** via `.env` file.

### Frontend

```bash
cd fiori
npm run build
cf push
```

Uses `Staticfile` buildpack to serve the built UI5 app as a static site.

## Security

- **No credentials in source code** — all secrets loaded from environment variables via `python-dotenv`
- **`.env` excluded from git** — protected by `.gitignore` at root and `backend/` level
- **`sample.env` committed** — contains only empty placeholders, safe for version control
- **HANA connection** — TLS encrypted (`HANA_ENCRYPT=True`), certificate validation configurable
- **CORS** — configured in FastAPI; restrict `allow_origins` for production deployments
